# 密钥管理 (Secrets Management)

> "OpenClaw的密钥管理系统设计得太安全了。运行时快照机制确保密钥配置不可变，防止意外修改。多源密钥解析支持 env、file、exec 三种来源，每个都有严格的验证逻辑。卧槽，这个安全审计功能让每次密钥使用都有记录，对于合规性和故障排查非常有价值。"

---

## 核心技术洞察

### 1. 运行时快照机制

```typescript
// src/secrets/runtime.ts
export type SecretsRuntimeSnapshot = {
  env: Record<string, string>;
  file: Record<string, string>;
  exec: Record<string, SecretExecResult>;
  timestamp: number;
};

export function prepareSecretsRuntimeSnapshot(
  config: OpenClawConfig,
): SecretsRuntimeSnapshot {
  const snapshot: SecretsRuntimeSnapshot = {
    env: {},
    file: {},
    exec: {},
    timestamp: Date.now(),
  };

  // 解析环境变量密钥
  for (const [key, secretConfig] of Object.entries(config.secrets ?? {})) {
    if (secretConfig.type === "env") {
      snapshot.env[key] = resolveEnvSecret(secretConfig);
    }
    // ... 其他类型
  }

  return snapshot;
}
```

**Leon点评**：运行时快照机制提供了安全性保障：
1. **不可变性**：快照创建后不再修改，防止意外变更
2. **时间戳记录**：记录快照创建时间，便于审计
3. **按需解析**：只在需要时解析密钥，避免不必要的操作
4. **隔离性**：快照与配置分离，支持动态刷新

### 2. 多源密钥解析

```typescript
// src/secrets/resolve.ts
export async function resolveSecret(
  config: SecretConfig,
  snapshot: SecretsRuntimeSnapshot,
): Promise<string | undefined> {
  switch (config.type) {
    case "env":
      return resolveEnvSecret(config);

    case "file":
      return await resolveFileSecret(config);

    case "exec":
      return await resolveExecSecret(config, snapshot);

    default:
      throw new Error(`Unknown secret type: ${config.type}`);
  }
}

async function resolveExecSecret(
  config: SecretExecConfig,
  snapshot: SecretsRuntimeSnapshot,
): Promise<string | undefined> {
  // 检查缓存
  if (snapshot.exec[config.name]) {
    return snapshot.exec[config.name].value;
  }

  // 执行命令获取密钥
  const { stdout } = await exec(config.command, {
    env: process.env,
    timeout: config.timeout ?? 5000,
  });

  return stdout.trim();
}
```

**Leon点评**：多源密钥解析提供了灵活性：
1. **env**：从环境变量读取，适合容器化部署
2. **file**：从文件读取，适合持久化存储
3. **exec**：从命令输出读取，适合动态密钥（如 AWS CLI）
4. **缓存机制**：exec 类型支持缓存，避免重复执行

### 3. 安全审计

```typescript

// src/secrets/validate.ts
export function validateSecretConfig(
  config: SecretConfig,
): { valid: boolean; error?: string } {
  if (!config.name) {
    return { valid: false, error: "Secret name is required" };
  }

  switch (config.type) {
    case "env":
      return validateEnvSecretConfig(config);
    case "file":
      return validateFileSecretConfig(config);
    case "exec":
      return validateExecSecretConfig(config);
    default:
      return { valid: false, error: `Unknown type: ${config.type}` };
  }
}
```

## 附录B：Chat会话中文件读取安全

> **用户关注点**：在 Chat 会话中，LLM 可以通过 Read 工具读取文件内容，这可能导致包含 API_KEY 等敏感信息的文件被读取并暴露给 LLM。

### 一、现有安全机制

OpenClaw 实现了多层文件访问安全防护：

#### 1. 边界文件读取 (Boundary File Reading)


- 拒绝目录访问

- 强制文件大小限制

### 二、安全风险识别

#### 当前保护的边界
- ✅ 防止路径遍历（`../../../etc/passwd`）
- ✅ 防止符号链接逃逸
- ✅ 防止硬链接攻击
- ✅ 限制文件大小
- ✅ 外部内容包装（web、email、webhook）

#### **存在的安全缺口**

**关键问题**：边界系统仅防止**逃逸工作区**，但不阻止读取工作区**内部的敏感文件**。

**风险场景**：
```bash
# 用户工作区结构
~/.openclaw/workspace/
├── .env                    # ❌ 包含 API_KEY=xxx
├── config/credentials.json # ❌ 包含敏感凭证
├── scripts/deploy.sh       # ❌ 可能包含密钥
├── src/
│   └── main.ts
└── README.md
```

如果 LLM 通过 Read 工具请求读取 `.env` 或 `config/credentials.json`，当前机制**不会阻止**，因为这些文件在工作区边界内。

### 三、缓解策略建议

#### 策略1：敏感文件模式拒绝（推荐实施）

```typescript
// src/infra/fs-safe.ts (建议添加)
const SENSITIVE_FILE_PATTERNS = [
  /\.env$/i,
  /\.env\.[a-z]+$/i,  // .env.local, .env.production
  /credentials\.json$/i,
  /secrets?\.(json|yaml|yml|txt)$/i,
  /private\.key$/i,
  /\.pem$/i,
  /\.key$/i,
  /auth\.token$/i,
  /\.aws\/credentials$/i,
  /\.aws\/config$/i,
  /\/\.config\/.*\/credentials$/i,
];

async function readFileWithinRoot(params: {
  rootDir: string;
  relativePath: string;
  // ... 其他参数
}): Promise<SafeLocalReadResult> {
  // 现有边界检查...

  // 新增：敏感文件检查
  const normalizedPath = relativePath.replace(/\\/g, '/');
  for (const pattern of SENSITIVE_FILE_PATTERNS) {
    if (pattern.test(normalizedPath)) {
      return {
        ok: false,
        error: "Sensitive file pattern detected",
        code: "sensitive-file",
      };
    }
  }

  // 继续正常读取...
}
```

#### 策略2：内容检测与脱敏（深度防御）

```typescript
// src/infra/fs-safe.ts (建议添加)
const SECRET_PATTERN_DETECTIONS = [
  /API[_-]?KEY\s*[:=]\s*[A-Za-z0-9\-_]{20,}/gi,
  /SECRET[_-]?KEY\s*[:=]\s*[A-Za-z0-9\-_]{20,}/gi,
  /PASSWORD\s*[:=]\s*[A-Za-z0-9\-_]{8,}/gi,
  /TOKEN\s*[:=]\s*[A-Za-z0-9\-_]{20,}/gi,
  /Bearer\s+[A-Za-z0-9\-_\.]{20,}/gi,
];

async function readFileWithinRoot(params: {
  // ...
}): Promise<SafeLocalReadResult> {
  // 现有读取...

  let content = await fs.readFile(fullPath, 'utf-8');

  // 检测敏感模式
  const detected: string[] = [];
  for (const pattern of SECRET_PATTERN_DETECTIONS) {
    const matches = content.match(pattern);
    if (matches) {
      detected.push(...matches);
    }
  }

  if (detected.length > 0) {
    // 记录安全事件
    logSecurityEvent({
      type: 'secret-detected',
      path: relativePath,
      patterns: detected,
    });

    // 返回脱敏内容
    content = redactSecrets(content);
  }

  return { ok: true, content };
}
```

#### 策略3：配置化文件访问控制

```yaml
# config.yaml
```

### 四、临时缓解措施

在代码层面实施前，用户可以采取以下措施：

1. **分离敏感文件**：
   ```bash

   mv ~/.openclaw/workspace/.env ~/.openclaw/.env
   ```

2. **设置文件权限**：

3. **使用环境变量**：
   ```bash

   ```

4. **Git 忽略敏感文件**：
   ```bash

   credentials.*
   ```

### 五、结论

OpenClaw 已实现了完善的**边界安全**机制，但缺少**内容级**的安全过滤。建议：

1. **短期**：实施敏感文件模式拒绝（策略1）
2. **中期**：添加配置化文件访问控制（策略3）
3. **长期**：结合内容检测与脱敏（策略2）

这将形成纵深防御体系，确保即使 LLM 被诱导尝试读取敏感文件，系统也能有效阻止。

## 附录C：密钥管理方式与API_KEY泄露防护

> **常见误解**：使用 env/file/exec 三种密钥管理方式能防止 API_KEY 被 LLM 读取泄露吗？

### 一、核心结论

**不能直接规避**。这是一个关键的安全概念混淆：

```
密钥存储方式 (env/file/exec)  ≠  防止 LLM 读取文件内容
```

| 层面 | 机制 | 防护目标 |
|------|------|----------|
| **密钥管理** | env/file/exec | 控制密钥**如何存储和获取** |
| **文件读取** | Read 工具 | LLM 可以读取工作区内**任何文件** |

**Leon点评**：env/file/env 只是解决了"密钥放在哪里"的问题，但没有解决"谁能读取包含密钥的文件"的问题。这是两个不同的安全边界。

### 二、实际风险场景

即使你使用了 `env` 方式存储密钥：

```yaml
# config.yaml
secrets:
  providers:
    env-provider:
      source: env  # ✅ 密钥从环境变量读取
```

但如果工作区里有这些文件：

```bash
~/.openclaw/workspace/
├── .env                    # ❌ LLM 可读取: API_KEY=sk-xxx
├── deploy.sh               # ❌ LLM 可读取: export API_KEY="sk-xxx"
├── config/credentials.json # ❌ LLM 可读取: {"apiKey": "sk-xxx"}
├── .aws/credentials        # ❌ LLM 可读取: aws_access_key_id=xxx
└── src/main.ts
```

**LLM 仍然可以通过 Read 工具读取这些文件并获取其中的 API_KEY**。

### 三、三种密钥方式的安全特性对比

```typescript
// src/config/types.secrets.ts

// 1. env - 环境变量方式
type EnvSecretProviderConfig = {
  source: "env";
  allowlist?: string[];  // 可选：环境变量白名单
};

// 2. file - 文件读取方式
type FileSecretProviderConfig = {
  source: "file";
  path: string;
  mode?: "singleValue" | "json";
  timeoutMs?: number;
  maxBytes?: number;
};

// 3. exec - 命令执行方式
type ExecSecretProviderConfig = {
  source: "exec";
  command: string;
  args?: string[];
  timeoutMs?: number;
  maxOutputBytes?: number;
  trustedDirs?: string[];
};
```

| 方式 | 防止文件泄露 | 防止内存泄露 | 防止日志泄露 | 适用场景 |
|------|-------------|-------------|-------------|----------|
| **env** | ❌ 不防止 | ❌ 不防止 | ⚠️ 部分 | 容器化部署 |
| **file** | ❌ 不防止 | ❌ 不防止 | ❌ 不防止 | 持久化存储 |
| **exec** | ❌ 不防止 | ⚠️ 动态获取 | ❌ 不防止 | 动态密钥 |

### 四、当前 OpenClaw 的防护边界

```
✅ 防止: ../../../etc/passwd           (路径逃逸)
✅ 防止: /root/.ssh/id_rsa             (工作区外)
✅ 防止: 硬链接攻击
✅ 防止: 符号链接逃逸
❌ 不阻止: .env                        (工作区内)
❌ 不阻止: config/credentials.json     (工作区内)
❌ 不阻止: scripts/deploy.sh           (工作区内)
```

**关键洞察**：边界系统只保护**工作区边界**，不保护**内容安全**。

### 五、真正的缓解方案

#### 方案1：分离敏感目录（推荐）

```bash
# 核心原则：包含 API_KEY 的文件绝不在工作区
~/.openclaw/
├── workspace/              # ✅ 工作区：无敏感文件
│   ├── src/
│   └── README.md
├── .env                    # ✅ 密钥文件：工作区外
├── credentials/            # ✅ 密钥目录：工作区外
│   └── api-keys.json
└── config/
```

#### 方案2：环境变量优先

```bash
# ❌ 不推荐：存储在 .env 文件
echo 'API_KEY=sk-xxx' > ~/.openclaw/workspace/.env

# ✅ 推荐：直接设置环境变量
export ANTHROPIC_API_KEY="sk-xxx"
export OPENAI_API_KEY="sk-yyy"

# ✅ 或者使用 shell 配置文件
echo 'export ANTHROPIC_API_KEY="sk-xxx"' >> ~/.zshrc
```

#### 方案3：文件权限 + Git 忽略

```bash
# 严格文件权限
chmod 600 ~/.openclaw/.env
chmod 600 ~/.openclaw/credentials/*.json

# .gitignore 确保不被提交
cat > ~/.openclaw/workspace/.gitignore << 'EOF'
.env
.env.*
*.key
credentials.*
*.pem
.aws/
EOF
```

#### 方案4：使用密钥管理工具（最佳实践）

```bash
# 使用 1Password CLI
op read 'op://Private/OpenAI/api-key' > ~/.openclaw/.env

# 使用 AWS Secrets Manager
aws secretsmanager get-secret-value --secret-id api-key

# 使用 Vault
vault kv get -field=api-key secret/openai
```

### 六、配置示例：安全的密钥管理

```yaml
# ~/.openclaw/config.yaml
secrets:
  providers:
    # 使用环境变量（最安全）
    env-provider:
      source: env
      allowlist:
        - ANTHROPIC_API_KEY
        - OPENAI_API_KEY

    # 如果必须用文件，确保文件在工作区外
    file-provider:
      source: file
      path: ~/.openclaw/credentials/secrets.json  # ⚠️ 不在 workspace/
      mode: json
      maxBytes: 10240

  defaults:
    env: env-provider
    file: file-provider
```

### 七、总结

| 问题 | 答案 |
|------|------|
| env/file/exec 能防止 LLM 读取 API_KEY 吗？ | **不能** |
| 它们的作用是什么？ | 控制密钥**如何存储**，不控制**谁能读取** |
| 当前如何规避？ | 1. 分离敏感文件到工作区外<br>2. 使用环境变量而非文件<br>3. 设置严格文件权限 |
| 最安全的做法？ | 将包含 API_KEY 的文件**完全移出工作区** (`~/.openclaw/workspace/`) |

**安全原则**：
> "包含密钥的文件不应该出现在 LLM 可以访问的任何地方。"
