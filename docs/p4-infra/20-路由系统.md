# 路由系统 (Routing System)

> "OpenClaw的路由系统设计得太优雅了。七层匹配优先级——从精确的 peer 匹配到兜底的 default fallback，每一层都有明确的职责。卧槽，这个 WeakMap 缓存设计太聪明了，当配置变更时自动失效，不会导致内存泄漏。而且线程消息的父级继承机制让我想到了 DOM 事件冒泡，这个设计在消息路由中简直是神来之笔。"

---

## 核心技术洞察

### 1. 七层匹配优先级：从精确到兜底

```typescript
// src/routing/resolve-route.ts
export function resolveAgentRoute(input: ResolveAgentRouteInput): ResolvedAgentRoute {
  const tiers: Array<{
    matchedBy: Exclude<ResolvedAgentRoute["matchedBy"], "default">;
    enabled: boolean;
    scopePeer: RoutePeer | null;
    candidates: EvaluatedBinding[];
    predicate: (candidate: EvaluatedBinding) => boolean;
  }> = [
    // 第1层：精确 peer 匹配（最高优先级）
    {
      matchedBy: "binding.peer",
      enabled: Boolean(peer),
      scopePeer: peer,
      candidates: collectPeerIndexedBindings(bindingsIndex, peer),
      predicate: (candidate) => candidate.match.peer.state === "valid",
    },
    // 第2层：线程父 peer 匹配（继承）
    {
      matchedBy: "binding.peer.parent",
      enabled: Boolean(parentPeer && parentPeer.id),
      scopePeer: parentPeer,
      candidates: collectPeerIndexedBindings(bindingsIndex, parentPeer),
      predicate: (candidate) => candidate.match.peer.state === "valid",
    },
    // 第3层：Guild + Roles 匹配
    {
      matchedBy: "binding.guild+roles",
      enabled: Boolean(guildId && memberRoleIds.length > 0),
      scopePeer: peer,
      candidates: guildId ? (bindingsIndex.byGuildWithRoles.get(guildId) ?? []) : [],
      predicate: (candidate) =>
        hasGuildConstraint(candidate.match) && hasRolesConstraint(candidate.match),
    },
    // 第4层：Guild 匹配（无角色）
    {
      matchedBy: "binding.guild",
      enabled: Boolean(guildId),
      scopePeer: peer,
      candidates: guildId ? (bindingsIndex.byGuild.get(guildId) ?? []) : [],
      predicate: (candidate) =>
        hasGuildConstraint(candidate.match) && !hasRolesConstraint(candidate.match),
    },
    // 第5层：Team 匹配
    {
      matchedBy: "binding.team",
      enabled: Boolean(teamId),
      scopePeer: peer,
      candidates: teamId ? (bindingsIndex.byTeam.get(teamId) ?? []) : [],
      predicate: (candidate) => hasTeamConstraint(candidate.match),
    },
    // 第6层：账户匹配
    {
      matchedBy: "binding.account",
      enabled: true,
      scopePeer: peer,
      candidates: bindingsIndex.byAccount,
      predicate: (candidate) => candidate.match.accountPattern !== "*",
    },
    // 第7层：渠道匹配（兜底）
    {
      matchedBy: "binding.channel",
      enabled: true,
      scopePeer: peer,
      candidates: bindingsIndex.byChannel,
      predicate: (candidate) => candidate.match.accountPattern === "*",
    },
  ];

  // 按优先级遍历，找到第一个匹配
  for (const tier of tiers) {
    if (!tier.enabled) {
      continue;
    }
    const matched = tier.candidates.find(
      (candidate) =>
        tier.predicate(candidate) &&
        matchesBindingScope(candidate.match, { ...baseScope, peer: tier.scopePeer }),
    );
    if (matched) {
      return choose(matched.binding.agentId, tier.matchedBy);
    }
  }

  // 兜底：使用默认 agent
  return choose(resolveDefaultAgentId(input.cfg), "default");
}
```

**Leon点评**：这个七层匹配设计太精妙了：
1. **peer 匹配**：最精确，针对特定用户/群组
2. **parent peer 继承**：线程消息继承父级匹配
3. **guild+roles**：Discard 角色匹配（需要任意一个角色匹配）
4. **guild**：Discard 服务器匹配（无角色限制）
5. **team**：Slack 工作区匹配
6. **account**：账户级匹配（跨 peer）
7. **channel**：渠道级兜底（`accountPattern: *`）

这种设计让路由既灵活又可预测。每个层次都有明确的职责，不会出现歧义。

### 2. WeakMap 缓存：配置变更自动失效

```typescript
// src/routing/resolve-route.ts
const resolvedRouteCacheByCfg = new WeakMap<
  OpenClawConfig,
  {
    bindingsRef: OpenClawConfig["bindings"];
    agentsRef: OpenClawConfig["agents"];
    sessionRef: OpenClawConfig["session"];
    byKey: Map<string, ResolvedAgentRoute>;
  }
>();

function resolveRouteCacheForConfig(cfg: OpenClawConfig): Map<string, ResolvedAgentRoute> {
  const existing = resolvedRouteCacheByCfg.get(cfg);
  // 检查引用是否变更（配置是否被修改）
  if (
    existing &&
    existing.bindingsRef === cfg.bindings &&
    existing.agentsRef === cfg.agents &&
    existing.sessionRef === cfg.session
  ) {
    return existing.byKey;  // 缓存有效
  }

  // 配置已变更，创建新缓存
  const byKey = new Map<string, ResolvedAgentRoute>();
  resolvedRouteCacheByCfg.set(cfg, {
    bindingsRef: cfg.bindings,
    agentsRef: cfg.agents,
    sessionRef: cfg.session,
    byKey,
  });
  return byKey;
}
```

**Leon点评**：这个 WeakMap 缓存设计太聪明了：
1. **自动失效**：当配置对象被替换时，旧缓存自动被 GC 回收
2. **引用比较**：通过比较 `bindingsRef`、`agentsRef`、`sessionRef` 来检测配置变更
3. **内存安全**：WeakMap 不会阻止配置对象被 GC，即使缓存还在引用
4. **性能优化**：避免每次路由都重新计算绑定

这种设计比普通的 Map + 手动清理要优雅得多，完全利用了 WeakMap 的语义。

### 3. DM 会话作用域：四种隔离级别

```typescript
// src/routing/session-key.ts
export function buildAgentPeerSessionKey(params: {
  agentId: string;
  channel: string;
  accountId?: string | null;
  peerId?: string | null;
  dmScope?: "main" | "per-peer" | "per-channel-peer" | "per-account-channel-peer";
}): string {
  const dmScope = params.dmScope ?? "main";

  // main：所有 DM 共享主会话
  if (dmScope === "main") {
    return buildAgentMainSessionKey({
      agentId: params.agentId,
      mainKey: DEFAULT_MAIN_KEY,
    });
  }

  // per-peer：每个用户一个会话（跨渠道）
  if (dmScope === "per-peer" && peerId) {
    return `agent:${agentId}:direct:${peerId}`;
  }

  // per-channel-peer：每个渠道的用户一个会话
  if (dmScope === "per-channel-peer" && peerId) {
    return `agent:${agentId}:${channel}:direct:${peerId}`;
  }

  // per-account-channel-peer：每个账户的渠道用户一个会话
  if (dmScope === "per-account-channel-peer" && peerId) {
    return `agent:${agentId}:${channel}:${accountId}:direct:${peerId}`;
  }
}
```

**Leon点评**：DM 会话作用域提供了四种隔离级别，满足不同场景：
1. **main**：所有 DM 共享一个会话（最简单，但可能混乱）
2. **per-peer**：每个用户独立会话（跨渠道，适合多渠道用户）
3. **per-channel-peer**：每个渠道的用户独立会话（适合同一用户在不同渠道有不同上下文）
4. **per-account-channel-peer**：每个账户的渠道用户独立会话（最精细，适合多账户场景）

这种设计让用户可以根据自己的需求选择会话隔离级别，非常灵活。

### 4. 身份链接解析：跨渠道用户识别

```typescript
// src/routing/session-key.ts
function resolveLinkedPeerId(params: {
  identityLinks?: Record<string, string[]>;
  channel: string;
  peerId: string;
}): string | null {
  const identityLinks = params.identityLinks;
  if (!identityLinks) {
    return null;
  }

  // 收集候选 ID
  const candidates = new Set<string>();
  const rawCandidate = normalizeToken(peerId);
  if (rawCandidate) {
    candidates.add(rawCandidate);  // 原始 peerId
  }

  // 渠道限定候选（channel:peerId）
  const channel = normalizeToken(params.channel);
  if (channel) {
    const scopedCandidate = normalizeToken(`${channel}:${peerId}`);
    if (scopedCandidate) {
      candidates.add(scopedCandidate);
    }
  }

  // 在身份链接中查找
  for (const [canonical, ids] of Object.entries(identityLinks)) {
    const canonicalName = canonical.trim();
    if (!canonicalName || !Array.isArray(ids)) {
      continue;
    }
    for (const id of ids) {
      const normalized = normalizeToken(id);
      if (normalized && candidates.has(normalized)) {
        return canonicalName;  // 返回规范化的用户名
      }
    }
  }

  return null;
}
```

**Leon点评**：身份链接解析解决了跨渠道用户识别的问题：
1. **配置格式**：`{ "canonical_user": ["telegram:123", "whatsapp:+123"] }`
2. **双候选策略**：同时匹配原始 peerId 和渠道限定的 peerId
3. **规范化返回**：返回配置中的规范用户名，而不是原始输入

这种设计让 OpenClaw 能够识别同一用户在不同渠道的身份，实现跨渠道的会话连续性。

### 5. 账户 ID 规范化：安全且可缓存的清理

```typescript
// src/routing/account-id.ts
const VALID_ID_RE = /^[a-z0-9][a-z0-9_-]{0,63}$/i;
const INVALID_CHARS_RE = /[^a-z0-9_-]+/g;
const LEADING_DASH_RE = /^-+/;
const TRAILING_DASH_RE = /-+$/;
const ACCOUNT_ID_CACHE_MAX = 512;

const normalizeAccountIdCache = new Map<string, string>();

function canonicalizeAccountId(value: string): string {
  // 如果已经符合规范，直接返回小写
  if (VALID_ID_RE.test(value)) {
    return value.toLowerCase();
  }
  // 否则清理：无效字符替换为 "-"，移除前后 "-"
  return value
    .toLowerCase()
    .replace(INVALID_CHARS_RE, "-")
    .replace(LEADING_DASH_RE, "")
    .replace(TRAILING_DASH_RE, "")
    .slice(0, 64);  // 限制长度
}

export function normalizeAccountId(value: string | undefined | null): string {
  const trimmed = (value ?? "").trim();
  if (!trimmed) {
    return DEFAULT_ACCOUNT_ID;  // "default"
  }

  // 缓存查找
  const cached = normalizeAccountIdCache.get(trimmed);
  if (cached) {
    return cached;
  }

  const normalized = normalizeCanonicalAccountId(trimmed) || DEFAULT_ACCOUNT_ID;
  setNormalizeCache(normalizeAccountIdCache, trimmed, normalized);
  return normalized;
}
```

**Leon点评**：账户 ID 规范化设计得很周到：
1. **快速路径**：如果已经符合规范，直接返回小写，避免不必要的处理
2. **安全清理**：替换无效字符，移除前后的 `-`，限制长度为 64
3. **原型污染防护**：调用 `isBlockedObjectKey` 检查（在 `normalizeCanonicalAccountId` 中）
4. **缓存优化**：最多缓存 512 个结果，超过后删除最旧的条目
5. **默认值**：空字符串返回 `"default"`，而不是 `undefined`

这种设计确保了账户 ID 既安全又高效。

### 6. 绑定索引：多维快速查找

```typescript
// src/routing/resolve-route.ts
type EvaluatedBindingsIndex = {
  byPeer: Map<string, EvaluatedBinding[]>;
  byGuildWithRoles: Map<string, EvaluatedBinding[]>;
  byGuild: Map<string, EvaluatedBinding[]>;
  byTeam: Map<string, EvaluatedBinding[]>;
  byAccount: EvaluatedBinding[];
  byChannel: EvaluatedBinding[];
};

function buildEvaluatedBindingsIndex(bindings: EvaluatedBinding[]): EvaluatedBindingsIndex {
  const byPeer = new Map<string, EvaluatedBinding[]>();
  const byGuildWithRoles = new Map<string, EvaluatedBinding[]>();
  const byGuild = new Map<string, EvaluatedBinding[]>();
  const byTeam = new Map<string, EvaluatedBinding[]>();
  const byAccount: EvaluatedBinding[] = [];
  const byChannel: EvaluatedBinding[] = [];

  for (const binding of bindings) {
    // Peer 索引（支持 group/channel 互换）
    if (binding.match.peer.state === "valid") {
      for (const key of peerLookupKeys(binding.match.peer.kind, binding.match.peer.id)) {
        pushToIndexMap(byPeer, key, binding);
      }
      continue;
    }

    // Guild + Roles 索引
    if (binding.match.guildId && binding.match.roles) {
      pushToIndexMap(byGuildWithRoles, binding.match.guildId, binding);
      continue;
    }

    // Guild 索引（无角色）
    if (binding.match.guildId && !binding.match.roles) {
      pushToIndexMap(byGuild, binding.match.guildId, binding);
      continue;
    }

    // Team 索引
    if (binding.match.teamId) {
      pushToIndexMap(byTeam, binding.match.teamId, binding);
      continue;
    }

    // Account 索引（非通配符）
    if (binding.match.accountPattern !== "*") {
      byAccount.push(binding);
      continue;
    }

    // Channel 索引（通配符）
    byChannel.push(binding);
  }

  return { byPeer, byGuildWithRoles, byGuild, byTeam, byAccount, byChannel };
}

function peerLookupKeys(kind: ChatType, id: string): string[] {
  if (kind === "group") {
    return [`group:${id}`, `channel:${id}`];  // group 和 channel 互换
  }
  if (kind === "channel") {
    return [`channel:${id}`, `group:${id}`];  // channel 和 group 互换
  }
  return [`${kind}:${id}`];
}
```

**Leon点评**：绑定索引设计支持多维快速查找：
1. **byPeer**：按 peer ID 精确查找（支持 group/channel 互换）
2. **byGuildWithRoles**：按 guild + roles 查找
3. **byGuild**：按 guild 查找（无角色限制）
4. **byTeam**：按 team 查找
5. **byAccount**：按账户模式查找
6. **byChannel**：渠道级通配符

这种索引设计让路由匹配从 O(n) 降到 O(1)，对于大量绑定配置的性能提升非常明显。

```typescript
function normalizeBindingMatch(
  match:
    | {
        accountId?: string;
        peer?: { kind?: string; id?: string };
        guildId?: string;
        teamId?: string;
        roles?: string[];
      }
    | undefined,
): NormalizedBindingMatch {
  return {
    accountPattern: (match?.accountId ?? "").trim(),
    peer: normalizePeerConstraint(match?.peer),
    guildId: normalizeId(match?.guildId) || null,
    teamId: normalizeId(match?.teamId) || null,
    roles: Array.isArray(match?.roles) && match.roles.length > 0 ? match.roles : null,
  };
}
```

## 五、关键技术权衡

### 1. 七层匹配 vs 单层匹配

| 方案 | 优势 | 劣势 |
|------|------|------|
| 七层匹配 | 灵活、可预测 | 复杂度高 |
| 单层匹配 | 简单 | 灵活性差 |

**选择**：七层匹配
**原因**：需要支持多种匹配场景（peer、guild、team、account），清晰的优先级确保可预测性

### 2. WeakMap vs 普通 Map

| 方案 | 优势 | 劣势 |
|------|------|------|
| WeakMap | 自动失效、内存安全 | 无法遍历 |
| 普通 Map | 可遍历、功能完整 | 需要手动清理 |

**选择**：WeakMap
**原因**：配置对象可能被替换，WeakMap 自动失效避免内存泄漏

### 3. 索引 vs 线性搜索

| 方案 | 优势 | 劣势 |
|------|------|------|
| 多维索引 | O(1) 查找 | 占用更多内存 |
| 线性搜索 | 内存占用小 | O(n) 查找 |

**选择**：多维索引
**原因**：路由是高频操作，性能优化更重要

### 4. DM 作用域 vs 单一模式

| 方案 | 优势 | 劣势 |
|------|------|------|
| 四种作用域 | 灵活、满足多种场景 | 配置复杂 |
| 单一模式 | 简单 | 无法满足所有需求 |

**选择**：四种作用域
**原因**：不同用户有不同的会话隔离需求

## 六、路由系统与其他模块的关系

**Q：路由系统和会话管理有什么关系？**

A：路由系统生成的 `sessionKey` 被会话管理系统用于标识和存储会话。同一个 `sessionKey` 的消息会被路由到同一个会话。

**Q：WeakMap 缓存什么时候会失效？**

A：当配置对象的 `bindings`、`agents` 或 `session` 引用发生变化时，缓存会自动失效。WeakMap 的键是配置对象本身，当旧配置被 GC 后，缓存也会被清理。

**Q：线程消息的父级继承是如何工作的？**

A：如果线程消息的 peer 没有匹配到绑定，系统会尝试使用父 peer（频道或群组）的绑定。这类似于 DOM 事件冒泡，子元素不处理时交给父元素处理。

**Q：身份链接如何工作？**

A：`identityLinks` 配置将多个渠道的用户 ID 映射到同一个规范名称。当生成会话键时，系统会查找身份链接并使用规范名称而不是原始 ID。

**Q：为什么 group 和 channel 可以互换？**

A：某些平台（如 Discord）对群组和频道的分类不明确，OpenClaw 通过允许互换来提高兼容性。`peerLookupKeys` 函数生成两个键来支持这种互换。

# config.yaml
bindings:
  # 1. 精确用户匹配（最高优先级）
  - agentId: vip-agent

  - agentId: moderator-agent

        - "moderator"
        - "admin"

  # 3. Slack 工作区匹配
  - agentId: slack-support
    match:
      channel: slack
      teamId: "T123456"

  # 4. 账户级匹配
  - agentId: work-agent
    match:
      channel: telegram
      accountId: "work-account"

  # 5. 渠道级兜底（最低优先级）
  - agentId: default-agent
    match:
      channel: telegram
      accountId: "*"

session:
  dmScope: per-channel-peer
  identityLinks:
    "alice":
      - "telegram:123456789"
      - "whatsapp:+1234567890"

## 附录B：会话键示例

```typescript
// 主会话
buildAgentMainSessionKey({ agentId: "main", mainKey: "main" });
// "agent:main:main"

// DM 会话（main 作用域）
buildAgentPeerSessionKey({
  agentId: "main",
  channel: "telegram",
  accountId: "default",
  peer: { kind: "direct", id: "123" },
  dmScope: "main",
});
// "agent:main:main"

// DM 会话（per-peer 作用域）
buildAgentPeerSessionKey({
  agentId: "main",
  channel: "telegram",
  accountId: "default",
  peer: { kind: "direct", id: "123" },
  dmScope: "per-peer",
});
// "agent:main:direct:123"

// 群组会话
buildAgentPeerSessionKey({
  agentId: "main",
  channel: "telegram",
  accountId: "default",
  peer: { kind: "group", id: "-100123" },
});
// "agent:main:telegram:group:-100123"

// 线程会话
resolveThreadSessionKeys({
  baseSessionKey: "agent:main:telegram:group:-100123",
  threadId: "456",
  useSuffix: true,
});
// { sessionKey: "agent:main:telegram:group:-100123:thread:456" }
```
